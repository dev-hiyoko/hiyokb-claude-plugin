# ops（運用・インフラ・リリース・障害対応）

**適用**: デプロイ/設定変更/障害対応など、本番に影響しうる作業。

## 手順
- [ ] 0. ドシエ確認（前回の進捗ログ・決定事項を読み再開 / 初回は対象と背景を把握）
- [ ] 1. 目的と影響範囲・リスクの確認（DoD: 何に影響しうるかを列挙）
- [ ] 2. 手順とロールバック策の用意（DoD: 失敗時の戻し方を決定事項に）
- [ ] 3. 事前告知・承認（DoD: 関係者へ周知、必要な承認取得）
- [ ] 4. 実施（DoD: 手順通り実行・各ステップの結果を進捗ログに）
- [ ] 5. 監視・確認（DoD: 指標/ログで正常性を確認）
- [ ] 6. 振り返り（学びを KB に。障害ならポストモーテム）

## Tips / 落とし穴
- ロールバック策のない変更を本番に入れない。
- 破壊的操作は実行前確認。
- 障害対応では「復旧」を最優先し、原因究明は復旧後に。
