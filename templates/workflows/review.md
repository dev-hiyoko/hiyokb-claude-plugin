# review（レビュー・PRレビュー）

**適用**: 他者/自分の変更を評価する。

> 成果物の置き場: review タスクのドシエは「実装方針」節を使わず、**所見は調査メモ、判断は決定事項**に残す。

## 手順
- [ ] 0. ドシエ確認（前回の進捗ログ・決定事項を読み再開 / 初回は変更の背景を把握）
- [ ] 1. 変更の意図と範囲を把握（DoD: 何を解決する変更か要約）
- [ ] 2. 機械的観点を自動化に委譲（DoD: `/code-review`・`/security-review` 等を先に通し、結果を調査メモに）
- [ ] 3. 正しさの確認（DoD: ロジック/境界/エラー処理を確認、観点を残す）
- [ ] 4. 設計・可読性・再利用の確認（DoD: 重複/過剰/命名を確認）
- [ ] 5. テスト・影響範囲の確認（DoD: テストの妥当性と抜けを確認）
- [ ] 6. フィードバック（DoD: 重大度 must/want/nits を分けて指摘 / 良い点も）
- [ ] 7. 振り返り（再発する観点はチェックリスト化して KB へ）

## Tips / 落とし穴
- **機械的観点（スタイル・既知の脆弱性パターン）は `/code-review`・`/security-review` に委譲し、人は設計と正しさに集中**（手順2と一貫）。
- 重大度（must / want / nits）を分けて伝える。
- 指摘は「なぜ」と根拠/代替案つきで。
